1. **P0 腳本：修正寬鬆 TLS 仍出現憑證鏈錯誤**（`packages/db-core`）  
   - **`pg` 合併設定**：`ConnectionParameters` 對含 **`connectionString`** 的設定使用 **`Object.assign({}, config, parse(connectionString))`**，URI 解析出的 **`ssl`** 會**覆寫**外層傳入的 **`ssl: { rejectUnauthorized: false }`**。  
   - **現象**：`RAILWAY_DATABASE_URL` 含 **`?sslmode=require`** 時，即使預設寬鬆 TLS，仍報 **`self-signed certificate in certificate chain`**；錯誤提示「設 **`RAILWAY_TLS_STRICT=0`**」在此情境**無效**（非環境變數問題）。  
   - **處理**：在**非** **`RAILWAY_TLS_STRICT`** 時，先從連線字串移除 **`sslmode`／`ssl`／`uselibpqcompat`**（**`stripSslQueryParams`**），再以明確的 **`ssl: { rejectUnauthorized: false }`** 連線；**保留帳密**（以 **`http:` 替換 scheme 後 `URL` 解析，再還原為 `postgresql:`**）。

12. **雲端／正式站：使用者主檔讀不到資料（`[nxui_base_user_list] 403 Permission denied`）**（`apps/nx-api`）  
   - **現象**：Vercel **nx-ui** 開啟「使用者主檔」時列表空白，紅條顯示 **`nxui_base_user_list` 403**。  
   - **根因**：**`UserController`** 全控制器 **`@Roles('ADMIN')`**；**DEMO-LITE／DEMO-PLUS** 種子僅 **`admin`** 帳號具 **`ADMIN`**，其餘展示帳號（如 **`chen_boss`** 為 **`OWNER`**）呼叫 **`GET /user`** 時 **`RolesGuard`** 回 **`Permission denied`**。與 CORS、DB 連線無關。  
   - **解法（使用者）**：讀寫分權——**`GET /user`**、**`GET /user/:id`** 改為種子六職務（後集中為 **`NX00_TENANT_MASTER_READ_ROLES`**）；**`POST`／`PUT`／`PATCH`** 維持僅 **`ADMIN`**。**`UserService.list`／`get`** 依 JWT **`tenantId`** 篩本租戶（平台 ADMIN **`tenantId` null** 不篩）。  
   - **驗收**：**`pnpm --filter nx-api run build`** 通過；具種子六職務之租戶帳號應皆可載入同租戶使用者列表（後續見同點 **補記**）。  
   - **涉及檔案**：**`apps/nx-api/src/nx00/user/controllers/user.controller.ts`**、**`services/user.service.ts`**、**`dto/user.dto.ts`**（新增 **`UserReadScope`**）。
   - **補記（同日）**：正式站重整仍 **403** 時，常見兩因：（1）**Railway nx-api 尚未重新部署**含上述變更的映像；（2）登入帳號僅具 **SALES／WAREHOUSE** 等，上一版 **`GET /user`** 只開 **ADMIN+OWNER**。已再改為 **`USER_READ_ROLES`**（與種子 **ROLE_SPECS** 六角色一致），**凡具任一租戶職務即可讀**使用者列表／明細；**寫入**仍僅 **ADMIN**。部署後無需換帳即可讀列表（**`tenantId` 篩選**不變）。
   - **補記 2（雙 403／Network）**：**`BaseUserMasterView`** 以 **`Promise.all`** 同時呼叫 **`listUsers`（pageSize=500）** 與 **`listRoles`（pageSize=200）**，故 Console 可見兩筆對 **nx-api** 的 **403**，不一定只有 **`/user`**。若僅放寬 **`GET /user`** 而 **`GET /role`** 仍 **`@Roles('ADMIN')`**，整頁仍會失敗。已將 **`RoleController`** 的 **`GET /role`**、**`GET /role/:id`** 讀取權限與 **`UserController`** 對齊，並共用常數 **`NX00_TENANT_MASTER_READ_ROLES`**（**`apps/nx-api/src/nx00/constants/tenant-master-read-roles.ts`**）；**`RoleService.list`／`get`** 亦依 JWT **`tenantId`** 篩本租戶。

15. **使用者職務設定：指派成員 403（`nxui_base_user_role_assign`）**（`apps/nx-api`）  
   - **現象**：畫面為 **「指定職務／隸屬使用者」**（非植物）；**POST `/user-role`** 回 **403**。  
   - **根因**：**`UserRoleController`** 原整類 **`@Roles('ADMIN')`**，**負責人（OWNER）** 無法指派。  
   - **作法**：**`UserRoleController`** 改為僅 **`JwtAuthGuard`**；**GET** 與 **user／role** 相同使用 **`assertTenantScopedOrPlatformAdmin`** + **`list`／`get` 依 `tenantId` 篩選**；**POST／PATCH** 加 **`assertAdminOrOwnerManager`**（**`req.user.roles` 含 ADMIN 或 OWNER**，與 **JwtStrategy validate** 一致，避免再依 **RolesGuard** 重查 DB 造成與他頁相同之誤擋）。**`UserRoleActionContext`** 新增 **`actorTenantId`**，**assign／revoke／setPrimary／setActive** 內 **`assertActorTenant`** 確保租戶操作者僅能改同租戶（**`actorTenantId === null`** 仍表平台）。

14. **使用者主檔仍 403（`nxui_base_role_list`）— 改讀取授權策略**（`apps/nx-api`）  
   - **推測根因**：**`RolesGuard`** 另查 **`nx00_user_role`**；若雲端 DB 與登入 JWT 不同步、或列為 **inactive**，即使 JWT 有效仍可能 **`roleCodes` 為空**而 **403**（與 **`@Roles` 已列六職務**無關）。  
   - **作法**：**`GET /user`**、**`GET /user/:id`**、**`GET /role`**、**`GET /role/:id`** 不再依 **`RolesGuard` 的職務比對**（該四支 handler **不設 `@Roles`**，Guard 見無 required roles 即放行）；改於 controller 呼叫 **`assertTenantScopedOrPlatformAdmin(req.user)`**：**JWT 已有 `tenantId`** 即可讀（資料仍由 **`UserService`／`RoleService` 之 `tenantId` 篩選**）；**無租戶**時僅 **`req.user.roles` 含 `ADMIN`**（平台管理）可讀。**`POST`／`PUT`／`PATCH`** 仍 **`@Roles('ADMIN')`**。  
   - **檔案**：**`src/nx00/utils/assert-tenant-read-context.ts`**；**`user.controller.ts`／`role.controller.ts`**；移除未再引用之 **`tenant-master-read-roles.ts`**。

## 遇到的問題與解法

| 問題 | 解法 |
|------|------|
| 執行 P0 時 Node 印出 **`SECURITY WARNING`**：`sslmode` 的 **`require`／`prefer`／`verify-ca`** 在目前 **`pg`** 被當成接近 **`verify-full`**，未來大版本會改貼近 libpq；並建議 **`uselibpqcompat=true`** 等 | **屬函式庫行為變更預告**，與連線埠／URI 是否「貼錯」無直接關係。P0 以腳本處理寬鬆 TLS 後可先忽略；若需嚴格校驗憑證則設 **`RAILWAY_TLS_STRICT=1`** 並使用可信任之 CA／連線方式。 |
| **`self-signed certificate in certificate chain`**，且已預期寬鬆 TLS | **根因**：URI 的 **`sslmode=require`** 經 **`pg-connection-string` 解析後覆寫**手動傳入的 **`ssl`**，導致仍做嚴格憑證校驗。**解法**：P0 腳本在寬鬆模式下**移除** URL 上相關 SSL 查詢參數，**只**用明確的 **`rejectUnauthorized: false`**（僅開發／連通性驗證用）。 |
| **`prisma validate`**：**`Nx00BrandCodeRole`** 報 **one-to-one relation must use unique fields on the defining side** | **`Nx00PartBrand.brandCodeRole`** 為 optional **一對一**，Prisma 要求 FK **`partBrandId`** 需 **`@unique`**。已於 **`partBrandId`** 保留 **`@unique`**，並**未**再加 **`@@unique([tenantId, partBrandId])`**（避免與一對一規則重複）；**`tenantId` 與 `@@index([tenantId])`** 仍保留。 |
| 計畫將舊 migration 放在 **`prisma/migrations/_archive/`** | Prisma 會把 **`migrations/` 底下每一個子資料夾**都當成一支 migration，導致 **P3015**（缺 **`migration.sql`**）或 **status** 出現待套用 **`_archive`**。**解法**：舊鏈改存 **`prisma/_archive_migrations/`**（與 **`migrations`** 同層）；**`migrations/README.md`** 已說明。 |
| **`migrate deploy` MIG001** 於**本機已有舊表**的庫（曾跑過 init～舊鏈） | **`42P07`** **`relation "nx00_user" already exists`**，且 **P3018** 鎖住後續 deploy。**解法**：**`pnpm exec prisma migrate resolve --rolled-back "20260403130000_MIG001-mw1-baseline"`**，再 **`pnpm exec prisma migrate reset --force`**（或 **`DROP SCHEMA public CASCADE` + `CREATE SCHEMA public`**）後重跑 **deploy**／**seed**。**Railway 空庫**無此問題。 |
| **`prisma migrate diff`** 報 **`--to-schema-datamodel` was removed**（Prisma 7） | 改用 **`--to-schema prisma/schema.prisma`**（與 **`--from-empty`** 併用可產 baseline SQL）。 |
| **`packages/db-core`** 需連 **Railway** 做 **migrate／seed**，但 **`DATABASE_URL`** 仍指本機 | 將 **`DATABASE_URL`** 與 **`prisma.config.ts`** 會讀的 **`DIRECT_URL`** 一併設為 **Railway Public URI**（與 **`RAILWAY_DATABASE_URL`** 一致即可）；本機 Docker 連線改註解保留。**`apps/nx-api/.env`** 若也要打同一顆庫須自行同步。 |
| **`pnpm exec prisma db seed`**：**`P1011`**／**`TlsConnectionError`**／**`self-signed certificate in certificate chain`**（底層 **`@prisma/adapter-pg` + `pg.Pool`**） | **與 P0 同根因**：Pool 若只傳 **`connectionString`**，URL 的 **`sslmode=require`** 經解析會覆寫 **`ssl`**，仍做嚴格憑證校驗。**解法**：seed 使用 **`poolConfigFromDatabaseUrl`**（**`scripts/pgTlsPoolConfig.ts`**）建立 Pool。Node 對 **`sslmode`** 的 **SECURITY WARNING** 仍屬 **`pg-connection-string` 未來行為預告**，與此次修補無衝突。 |
| Railway **`nx-api`** build：**`EUNSUPPORTEDPROTOCOL`**／**`workspace:*`**（含 **Root Directory** 在 **`/apps/nx-api`** 時子目錄 **`npm install`**） | **解法**：**`apps/nx-api/Dockerfile`** 自 repo 根建置（**pnpm**、**`prisma generate`**、**`nx-api` build**）；Railway **Root Directory** 改 **`/`**，**Build／Start Command** 清空，**Dockerfile path** **`apps/nx-api/Dockerfile`**。根目錄 **`packageManager`／`.npmrc`** 仍利於本機與其他 CI。 |
| Docker **`nest build`** 大量 TS：**`Property 'nx00User' does not exist on type 'PrismaService'`** 等 | **`nx-api`** 自 **`db-core`** 匯入 **`PrismaClient`**；**`db-core/package.json`** 的 **`types`** 指向 **`dist/index.d.ts`**，僅 **`prisma generate`** 而無 **`dist`** 時型別不完整。**解法**：**generate** 後執行 **`pnpm exec tsc -p packages/db-core/tsconfig.json`**（**hoisted** 下避免用 **`pnpm --filter db-core run build`**，因子套件內可能找不到 **`tsc`**）。 |
| 正式站 **使用者主檔** **`nxui_base_user_list` 403**、列表空白 | 初因：**`GET /user`** 僅 **`ADMIN`**；續因：僅開 **ADMIN+OWNER** 時 **SALES／WAREHOUSE** 等仍 **403**；再因：頁面同時打 **`GET /role`（pageSize=200）**，**`RoleController`** 原僅 **ADMIN**，**Promise.all** 一失敗整頁掛。**解法**：**`/user`／`/role` 讀取**皆開放種子六職務（**`NX00_TENANT_MASTER_READ_ROLES`**）+ **`tenantId` 篩選**；**寫入**仍 **ADMIN**。若已改仍 **403**→ 查 Network 實際路徑、並確認 **Railway 已部署**。見 **第 12～13 點**。 |

### 17. 租戶資料隔離：租戶 admin 不再視為跨租戶平台＋使用者職務 UI 隱藏項

**現象（使用者職務設定）**：登入租戶 **admin** 時 JWT 將 **`tenantId` 清空**，**`GET /role`／`/user`／`/user-role`** 未篩租戶，三間公司職務與 **admin** 重複出現；無法分辨公司。

**根因**：登入與 **JwtStrategy** 將「具 **ADMIN** 職務」一律當成平台管理員。

**解法**：

- **跨租戶平台**僅當 **`nx00_user.tenant_id IS NULL`** 且具 **ADMIN**；**租戶內 admin**（**DEMO／DEV** 種子）JWT **保留該租戶 `tenantId`**，API 讀取範圍與 **`assertTenantScopedOrPlatformAdmin`** 一致。
- **`/auth/me`**：`roles`／**`view_permissions`** 合併時，有租戶者僅採**該租戶**之 **user_role**；**`view_permissions`** 之「略過矩陣」僅 **`tenantId == null` 且 ADMIN**。
- **JwtStrategy**：依 **`nx00_user.tenantId`** 篩 **user_role**；補齊 **`tenantCode`／`planCode`**（舊 token 可補救）。
- **租戶範圍內**：**`GET /role`** 不列出 **`code === 'ADMIN'`**；**`GET /user`** 與 **`GET /user-role`** 不列出 **`userAccount`（不分大小寫）為 `admin`**；**`assign`** 拒絕對 **admin** 帳號或 **ADMIN** 職務（**`actorTenantId` 有值**時）。

**驗證**：以 **DEMO-LITE admin** 登入後應僅見該租戶職務／使用者；**重新登入**取得新 JWT（舊 token 仍可由 validate 依 DB **`tenantId`** 修正範圍，但登入 payload 仍以新邏輯為準）。

### 18. ADMIN 職務：略過畫面權限 Guard＋職務權限矩陣後端鎖全開（2026-04-03）

- **現象**：租戶 **admin**（JWT 含 **`tenantId`**）若僅依 **`nx00_role_view`** 檢查，矩陣未齊或全關時會出現 **`nxui_base_user_list` 等 403**，與「管理員應能進主檔」預期不符。
- **作法**：
  - **`ViewPermissionService.mergeForRequestUser`**：請求使用者 **`roles` 含 `ADMIN`** 即回傳 **`null`**（略過 **Nx00ViewPermissionGuard**），**不再**要求 **`tenantId == null`**；**租戶資料範圍**仍只靠 JWT **`tenantId`** 與各 **Service** 篩選，不靠矩陣。
  - **`GET /auth/me`**：具 **ADMIN** 職務時 **`view_permissions` 為 `null`**（前端視為全開）。
  - **`RoleViewService`**（**`code === 'ADMIN'`**）：**`toDto`** 五維一律顯示全開；**`grant`／`updatePerms`／`upsertByRoleId`** 寫入強制五維全開；**`revoke`**、**`setActive(false)`** 拒絕（**BadRequest**：不可撤銷／不可停用）。
- **涉及檔案**：**`apps/nx-api/src/nx00/rbac/view-permission.service.ts`**、**`apps/nx-api/src/auth/services/auth.service.ts`**、**`apps/nx-api/src/nx00/role-view/services/role-view.service.ts`**。
- **驗收**：**`pnpm --filter nx-api run build`** 通過。
